# Аудит исходных файлов и выполненные исправления

Ниже перечислены исходные файлы пользователя и основные проблемы, которые были исправлены в собранной версии проекта.

## Backend

### `deepseek_go_20260408_245f95.go` → `main.go`
Проблемы:
- `logout` был оставлен среди публичных маршрутов, хотя логически требует авторизацию.
- сервер не раздавал HTML-страницы и статические файлы, хотя фронтенд был загружен как набор `html`-страниц.
- CORS был завязан только на `localhost:3000`, при этом фронтенд работал как обычные статические страницы.

Исправления:
- `logout` перенесен в защищенную группу `/api`.
- добавлена раздача страниц из каталога `web`.
- конфигурация CORS расширена под локальный запуск на `8080` и `3000`.

### `deepseek_go_20260408_9217e8.go` → `database/db.go`
Проблемы:
- использовалась функция `gen_random_uuid()`, но в `init.sql` подключалось только расширение `uuid-ossp`.
- отсутствовали таблицы `company_followers` и `community_join_requests`, хотя хендлеры активно к ним обращались.
- значения по умолчанию и массивы были заданы непоследовательно.
- ошибки создания таблиц не останавливали запуск надежным образом.

Исправления:
- добавлено расширение `pgcrypto`.
- добавлены отсутствующие таблицы.
- выровнены `DEFAULT`-значения для массивов и счетчиков.
- схема сделана самодостаточной для запуска.
- добавлена функция `EnsureDefaultCommunityRoles`.

### `deepseek_go_20260408_27660b.go` → `models/models.go`
Проблемы:
- отсутствовала модель `ForumReply`, хотя она использовалась в хендлере форума.
- часть структур была неполной для стабильной сериализации.

Исправления:
- добавлена `ForumReply`.
- выровнены структуры данных под реальный JSON-ответ API.

### `deepseek_go_20260408_e8363e.go` → `middleware/auth.go`
Проблемы:
- middleware проверял только подпись JWT, но полностью игнорировал таблицу `sessions`.
- после logout токен оставался рабочим до истечения срока действия.

Исправления:
- middleware теперь дополнительно проверяет наличие сессии в БД.
- logout действительно инвалидирует токен, удаляя запись сессии.

### `deepseek_go_20260408_864c65.go` → `utils/jwt.go`
Проблемы:
- срок жизни токена был захардкожен.
- не было защиты от пустого `JWT_SECRET`.
- проверка алгоритма подписи была слишком мягкой.

Исправления:
- срок действия считывается из `JWT_EXPIRE_HOURS`.
- добавлена защита от пустого секрета.
- усилена проверка метода подписи.

### `deepseek_go_20260408_0e53ab.go` → `handlers/auth.go`
Проблемы:
- `Logout` удалял из БД заголовок `Authorization` целиком, а не чистый токен.
- `Terms` в регистрации не валидировался реально как согласие.
- login/logout были связаны с таблицей `sessions` непоследовательно.

Исправления:
- logout работает с токеном из middleware.
- добавлена реальная проверка согласия с условиями.
- логика создания и удаления сессии выровнена.

### `deepseek_go_20260408_ce0ee4.go` → `handlers/dashboard.go`
Проблемы:
- `ProjectsCount` был просто зафиксирован как `3`.

Исправления:
- статистика теперь считается из данных.

### `deepseek_go_20260408_2a70ac.go` → `handlers/friends.go`
Проблемы:
- логика была в целом рабочей, но не все ошибки БД обрабатывались одинаково.
- требовалось привести ответы к единообразию.

Исправления:
- выровнена обработка ошибок.
- оставлена логика авто-принятия встречного запроса.

### `deepseek_go_20260408_516b2b.go` → `handlers/communities.go`
Проблемы:
- массив `search_tags` сканировался как `sql.NullString`, хотя в БД это `TEXT[]`.
- использовалась таблица `community_join_requests`, которой не было в схеме.
- `members_count` увеличивался/уменьшался без проверки, был риск рассинхронизации.
- роли по умолчанию не гарантировались для каждой новой записи.

Исправления:
- сканирование переведено на `pq.StringArray`.
- таблица добавлена в схему.
- счетчики пересчитываются по фактическим данным.
- при создании сообщества создаются дефолтные роли.

### `deepseek_go_20260408_81eba1.go` → `handlers/community_roles.go`
Проблемы:

- часть проверок прав доступа была нестабильной.

Исправления:
- перед чтением ролей обеспечивается наличие базового набора ролей.
- выровнены проверки прав.

### `deepseek_go_20260408_111434.go` → `handlers/companies.go`

- использовалась таблица `company_followers`, которой не было в схеме.

Исправления:
- таблица добавлена.

- счетчик подписчиков пересчитывается по фактическому составу таблицы.

### `deepseek_go_20260408_ab855e.go` → `handlers/company_roles.go`
Проблемы:
- `employee_count` менялся арифметически, что создавало риск накопления ошибки.

Исправления:

### `deepseek_go_20260408_a090d6.go` → `handlers/posts.go`
Проблемы:
- `SELECT p.*` использовался вместе со сканированием вручную, что делает код хрупким при изменении схемы.
- `likes_count` и `comments_count` изменялись без гарантии реального insert/delete.
- `target_id` и массивы тегов обрабатывались неустойчиво.

Исправления:
- список полей в `SELECT` сделан явным.
- счетчики пересчитываются по фактическим таблицам `post_likes` и `post_comments`.
- парсинг `target_id` и `tags` сделан безопаснее.

### `deepseek_go_20260408_395dd1.go` → `handlers/forum.go`
Проблемы:

- чтение ответов темы было неполным.

Исправления:

- выровнен запрос к ответам форума.

### `deepseek_go_20260408_6f7e22.go` → `handlers/chat.go`
Проблемы:
- сообщения чата можно было читать без проверки участия пользователя в чате.
- `last_time` читался в `sql.NullString`, хотя это `timestamp`.
- при чтении чата сбрасывались сообщения без жесткой проверки доступа.

Исправления:
- добавлена проверка участия в чате для чтения и отправки.
- `last_time` переведен на `sql.NullTime`.
- обнуление `unread_count` теперь выполняется только для участника чата.
